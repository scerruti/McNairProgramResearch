# Justifying the Phase‑1 Statistical Test Battery

This document explains and justifies the statistical tests implemented in `notebooks/phase1_stat_tests_cerruti.ipynb`. It is written for a student who wants to understand why each test is used, what it measures, how to interpret results, and what follow‑ups or caveats to keep in mind.

## Purpose

- High‑level goal: determine whether the Phase‑1 expert‑activation data contains structure beyond random noise — i.e., whether particular experts (or sets of experts) are associated with particular question categories, modalities (image vs text), or behaviours (correct vs incorrect).
- Practical motivation: identifying meaningful expert specialization (e.g., visual specialists vs text specialists) supports targeted analyses in subsequent phases (per‑layer analysis, ablation, clustering).

## Data & assumptions

- Input: a results JSON (default `/content/lego_moe_expert_analysis.json`) containing per‑question records.
- Each question record should include:
  - `category`: a categorical label (e.g., rotation, height)
  - one or more expert activation summaries, commonly `expert_frequency` (counts per expert). The notebook will also accept `image_expert_frequency` or `text_expert_frequency` if present.
- Counts are assumed to be aggregated activation counts (e.g., times an expert was selected across tokens/layers for a question).
- Important: question lengths (token counts) may vary and bias raw counts. Consider per‑question normalization (divide each question vector by its total activations) if that is a concern.

## Tests included and why

The notebook mixes descriptive metrics, contingency/association tests, information‑theoretic measures, and permutation tests. Each test answers a distinct question about the data.

1) Descriptive measures: entropy, normalized entropy, Gini coefficient
- Why: quantify how concentrated or diffuse expert usage is for each category.
- What: entropy measures spread; normalized entropy scales to [0,1]; Gini measures inequality (0 = equal, 1 = very unequal).
- Interpretation: low entropy / high Gini ⇒ specialization (few experts dominate). Compare across categories to find which are specialized vs generalist.
- Caveat: compare like‑with‑like (normalize for differing totals if needed).

2) Contingency analysis: chi‑square test + Cramer’s V
- Why: test whether expert usage depends on category.
- What: build an experts × categories contingency table and test independence. Cramer’s V gives effect size.
- Interpretation: significant chi‑square and non‑trivial Cramer’s V suggests dependence (some experts preferentially handle certain categories).
- Caveat: chi‑square reliability suffers when many expected counts are small. When sparse, complement with permutation tests.

3) Mutual Information (MI) with permutation test
- Why: measure how much expert id reduces uncertainty about category (captures nonlinear associations) and assess significance by permutation.
- What: expand counts into repeated label arrays (capped in size for memory), compute MI(category; expert), and estimate p‑value by label permutation.
- Interpretation: MI > 0 with small permutation p‑value indicates expert selection is informative about category.
- Caveat: expansion and capping (SAMPLE_MI_MAX) trades precision for memory safety; MI magnitude should be read alongside the p‑value.

4) Top‑K overlap permutation test
- Why: ask whether the top K experts for different categories overlap more than expected by chance.
- What: select top‑K experts for each category, compute pairwise intersections, and compare to a null generated by sampling random top‑K sets from the expert pool.
- Interpretation: unusually large overlap suggests shared processing needs or common expert usage across categories.
- Caveat: ignores counts inside top‑K (presence/absence only). Choice of K affects sensitivity.

5) Random‑split control (pairwise permutation of L2 distance)
- Why: test at the question level whether two categories have different aggregate expert profiles beyond chance.
- What: compute observed L2 norm between category aggregated vectors, then randomly split combined questions into groups of the same sizes many times to get a null distribution.
- Interpretation: small permutation p‑value implies categories differ more than a random split would produce.
- Strengths & caveats: preserves within‑question structure and is conservative; sensitive to group sizes and dominated by large differences in some experts.

## Why permutation tests?

- Many tests involve sparse, high‑dimensional counts that violate standard parametric assumptions. Permutation tests build an empirical null distribution by random reassignment and are appropriate for complex statistics (MI, top‑K overlap, L2 distance). Their cost is computational, so choose the number of permutations balancing precision and runtime.

## Statistical practice & interpretation guidance for students

- Always state the null hypothesis for each test (e.g., independence for chi‑square; no information for MI; random overlap for top‑K; no difference for the random split).
- Report both p‑values and effect sizes (Cramer’s V, MI magnitude, observed overlap). A small p‑value with a negligible effect size may not be practically meaningful.
- Correct for multiple comparisons where relevant (e.g., pairwise top‑K overlap p‑values across many category pairs) using FDR or Bonferroni.
- Consider per‑question normalization when question lengths vary: divide each question’s expert counts by its total activations before aggregation.
- Robustness checks:
  - Repeat analyses with normalized counts.
  - Exclude very rare experts (low overall activations) or collapse the tail.
  - Run the battery per layer if per‑layer data exists to find where specialization occurs.
  - Replicate the analysis on Phase‑2 (`data/phase2/runpod_second/results.json`).

## Practical settings & computational notes

- Default permutation counts: the notebook uses N_PERMUTATIONS=2000 for top‑K and random‑split tests. MI uses 500 permutations by default. Increase these for tighter p‑values at higher computational cost.
- Random seed: RNG is seeded for reproducibility in the notebook; keep or change the seed consistently if you re‑run.
- MI expansion: SAMPLE_MI_MAX limits memory usage (default 2,000,000). If you have ample memory, increase this cap for better MI precision.

## Example interpretations

- Cramer’s V = 0.35 and chi‑square p < 0.001: moderate association between expert usage and category — investigate which experts drive the effect.
- MI = 0.05 with permutation p = 0.02: expert id carries a small but significant amount of information about category — visualize to assess practical significance.
- Observed top‑K overlap between A and B = 8 with p = 0.001: A and B share more top experts than expected — inspect shared experts and question similarities.

## Suggested follow‑ups

- If your JSON contains `image_expert_frequency` and `text_expert_frequency`, run the battery separately for each to test modality specialization directly.
- Perform per‑layer analysis if `per_layer_experts` data is available to locate where in the model routing specialization emerges.
- Produce heatmaps of experts × categories and of per‑category top experts to support intuitive understanding of numeric results.
- Add multiple comparison correction for pairwise tests when reporting many p‑values.
